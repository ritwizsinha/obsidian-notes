##### Deep memory Retrieval
Is a bunch of concepts or benchmarks that tests agent's accuracy on long range recall problems
1. Needle in haystack
2. Long context recall
3. Memory hop reasoning

Zep provides a temporally aware knowledge graph
LongMemEval benchmark has a bunch of temporal reasoning tasks to benchmark on
This kinds of frameworks improve accuracy while at the same time reducing reasoning requirements and latency to the answer


### Knowledge Graph Structure

##### Episodes
These are atomic messages and context provided to the model through which entity extraction happens. The episode can contain the text (message, text or JSON) + actor + reference timestamp. There are two timestamps stored 
1. T: Chronological order of events
2. T': Order in which ingestion happened
The episodes extracted entities are connected through a bidirectional edge to the episodes itself.

#### Entities
Entities are extracted from the messages using named entity recognition across the last 4 messages to provide context. Speaker is automatically extracted as an entity. Each entity is then embedded into a 1024 dimensional vector, enabling cosine similarity search and we can also search by full text search. We then passed these extracted entities along with their episodes through an LLM and find out which entities to update and which to create new. Entity edges represent the relation between entities extracted. 

#### Temporal view of data
Each edge contains its t-valid and t-invalid timestamp, and edge is invalidated when a semantically similar edges negates the fact about the information in the edge, this is then added in the t-invalid timestamp.  This is done using an LLM

#### Communities
Communities are generated by creating a map reduce summarisation over entities and episodes.


### Search
For search there is a 3 step process
1. Collect relevant edges, entity nodes and community nodes
2. Rerank them using a reranker model
3. Convert these items into text context

For collecting relevant edges, nodes and community nodes we use 3 types of search
1. BM25 full text search
2. Cosine similarity search
3. BFS search

In search atleast we don't search using the temporal data, instead after step 3 and given the context, we return everything whether that is temporally valid or not, leaving the LLM to figure that out



### Benchmarks
The Deep Memory Retrieval benchmark problem is that each set of conversation sessions is only upto 60 messages which can easily fit into memory. This benchmark is considered highly underdeveloped mainly because of the high accuracy with the full conversation which is a result of very small datasets such that everything fits in context. LongMemEval is much better form of benchmark with much longer conversation sessions. representing real industry scenarios


This paper claims to be an improvement over MemGPT but wasn't able to replicate the results present in MemGPT moreover was not able to use MemGPT over the LongMemEval benchmark. Thus they just compared it with full conversation contexts in the model


### Conclusion
1. Fine tuned models for entity edge extraction could signifcantly improve the accuracy
2. Defining knowledge graph ontologies might be better while creating a KG

### Title: Zep a temporal knowledge graph architecture for agent memory

This is _Deep Dive Archive_
and today we’re breaking down one of the most talked-about papers in agent memory systems: **Zep a temporal knowledge graph architecture for agent memory.**
Agentic memory has exploded recently and for good reason.  
If agents are going to feel intelligent, they _must_ remember. Not just store data, but _use it meaningfully_ to personalize interactions.

Right now, most systems are basically stateless or rely on short-term context windows. That’s not enough.

The market is clearly moving toward agents that can:
- Remember past interactions
- Adapt to user behavior
- And evolve over time

And this paper proposes a really interesting architecture to make that happen.

Although it might seem a rather dense title but can be broken down into 2 main innovations
1. Knowledge Graph
2. Temporal Events

Knowledge graph can be considered as a dense graph representation of knowledge.
Just like a graph it has edges, nodes, clusters. 
The connection between nodes represents piece of information. 
Using LLMs, we can extract these nodes and edges directly from raw text and convert them into a **directed graph**.
This structure of knowledge is then highly useful when agents require information about some particular topic or user where we can extract the node, its edges and neighbours to provide a comprehensive recall of all the information we have for the topic.
I'll talk about detail about knowledge graphs in another video.

Let's categorise the kinds of queries which we expect our agent or LLM to answer for us:
1. Needle in haystack: we have huge corpora of text and information and we need some particular information for it. (BM25 FTS and cosine simlarity perform pretty well on it)
2. Long Range Recall: A generic kind of needle in a haystack issue where we need a piece of information from a very old context provided
3. Multi Hop reasoning: Requires combining multiple source facts
	1. Temporal Memory Tracking: Questins about how data/facts have changed over time


Each of these questions require specific search capabilities

1. Requires good full text search and cosine similarity search
2. Requires good incremental discovery 
3. Requires the memory storage to be connection aware thus graph
4. Requires timestamp indexed data store along with graph


Thus temporal events basically mean events which are ordered on the concept of time.

In this paper is every interaction deemed to be ingested to memory is ingested as episodes. 
Thus these episodes would be plain text/json. 
After ingesting the episode, we extract entities from it using a common entity extraction algorithm. 
This can be done using an LLM or a model meant for just entity extraction.
These entities are then resolved for duplicates and added to a vector store for similarity search requirements. 
After resolution these are incorporated in the knowledge graphs as nodes connected to the episode.

Facts are represented as the edges of these entities.
A fact would be a combination some text, relation type and some attributes. 
The most important innovation here is that they expire. 
Thus there might be some edges that are expired depending on how we are querying it by.
We also create a fact_embedding to be able to search for closely related facts. 
Facts thus contain a valid time and an invalid time. 
While resolution using an LLM we find facts contradicting each other we invalidate the previous flag and add an invalid time there and create a new fact or edge.


After the graph is created we do a community detection using some standard algorithm and try to summarise a community with a community node.

Now our knowledge base is created, for finding relevant data we create custom search techniques.

Search it employs a three step method which includes a combination of bfs search across the graph, a BM25 Full text search and a cosine similarity search.
This gives us similar items across 3 dimensions
1. Multi hop reasoning using bfs
2. Exact matches using Full text search
3. Semantic matches using vector search.

After we get a bunch of candidates using this search we have already optimized for recall now we improve precision using the reranker.
There are various forms of reranking algorithms like RRF and MMR.

They have tested this approach mainly on the DMR and the LongMemEval benchmark. 
These benchmarks provide a long chat session and based on the conversation ask questions from the model to test its memory.
For DMR this paper already considers it an inferior benchmark where there seems to be no need for memory because the chat sessions are really small enough that they can fill up context.
On the long mem eval benchmark however the model does show actual improvement in the number of tokens used and the latency, the accuracy of the model also increased compared to full conversation.

Overall this seems to be a pretty good memory architecture, some open questions though which I would keep a lookout for:
- The latency aspects of it which might increase as the number of components like full text search, BFS search and vector embedding search all increase load on the system in production environments.
- Another concern would be their use of LLMs in areas of memory update which might be time consuming, but since memory update can be asynchronous we can be good with that. 
- Also it might also be an option for the customer to provide an ontology instead of the LLM inferring it from the data, which will make search results more accurate.
